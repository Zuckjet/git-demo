## 一、为什么你需要了解 Git？

在日常工作中，你是否遇到过这些场景？

| 场景 | 痛点 |
|------|------|
| 文件改来改去，不知道哪个是最新版 | `需求文档_v1.docx` `需求文档_v2_final.docx` `需求文档_v2_final_真的最终版.docx` |
| 和同事协作同一份文档，互相覆盖 | "你把我改的内容覆盖掉了！" |
| 改错了，想回到之前的版本 | 发现 Ctrl+Z 已经不够用了 |
| 不知道这个文件是谁改的、改了什么 | 无从追溯，责任不清 |

**Git 就是解决这些问题的工具。** 它是一套"版本管理系统"，帮你记录文件的每一次变化，随时可以回溯、对比、协作。

## 二、Git 是什么？用一个生活比喻来理解

> 💡 **把 Git 想象成一个"超级存档系统"**

就像玩游戏时，你可以在任意时间点"存档"，之后无论走了多少弯路，都可以"读档"回到那个时间点。

- **每次存档** = `commit`（提交）
- **存档记录** = 版本历史
- **多个存档槽** = `branch`（分支）
- **云端备份** = 远程仓库（如 GitHub / GitLab）

---

## 三、核心概念（5 个关键词）

### 1. 📁 仓库（Repository / Repo）

存放项目所有文件和历史记录的地方，可以理解为一个"有记忆的文件夹"。

- **本地仓库**：在你自己电脑上的
- **远程仓库**：在服务器上的（如公司的 GitLab、或 GitHub）

---

### 2. 📸 提交（Commit）

对当前文件状态做一次"快照存档"，并附上说明。

> 类比：就像你在文档里写"2024-01-10 修改了第三章的需求描述"，Git 会帮你自动记录**谁**、**什么时间**、**改了什么**。

每次 commit 都有：
- 唯一的 ID（像身份证号）
- 提交说明（你写的备注）
- 作者和时间

---

### 3. 🌿 分支（Branch）

可以理解为"平行宇宙"——在不影响主线的情况下，开辟一条独立的工作线。

```
主分支 (main):  ●──●──●──●──●
                         \
新功能分支:               ●──●──●
```

> 类比：你在修改一份方案，但不确定新方案是否可行，所以复制一份来改，原版保持不动。分支就是这个"复制版"，做好了再合并回去。

---

### 4. 🔀 合并（Merge）

把一个分支的改动合并到另一个分支，就像把"修改好的复制版"内容合并回原版。

---

### 5. ⬆️⬇️ 推送与拉取（Push / Pull）

| 操作 | 含义 | 类比 |
|------|------|------|
| `Push`（推送）| 把本地的改动上传到远程仓库 | 把文件上传到云盘 |
| `Pull`（拉取）| 把远程仓库最新的内容下载到本地 | 从云盘同步最新文件 |

## 四、安装与环境配置

### 📥 Step 1：下载安装 Git

#### Windows 用户

1. 访问官网：[https://git-scm.com/download/win](https://git-scm.com/download/win)
2. 下载自动开始，运行安装包
3. 一路点 `Next`，**保持默认选项**即可
4. 安装完成后，在桌面或任意文件夹右键，能看到 `Git Bash Here` 说明安装成功 ✅

#### macOS 用户

1. 打开终端（Terminal），输入以下命令：
   ```bash
   git --version
   ```
2. 如果没有安装，系统会**自动弹出提示**，点击安装即可
3. 或者访问官网下载：[https://git-scm.com/download/mac](https://git-scm.com/download/mac)

#### 验证安装成功

打开终端（macOS）或 Git Bash（Windows），输入：

```bash
git --version
```

看到类似 `git version 2.43.0` 的输出，说明安装成功 ✅

---

### 👤 Step 2：配置身份信息（必做！）

安装完 Git 之后，**第一件事**就是告诉 Git 你是谁。这些信息会附在你每一次提交记录上，让团队知道是谁做了改动。

打开终端 / Git Bash，输入以下两条命令：

```bash
# 配置用户名（建议用真实姓名或英文名）
git config --global user.name "你的名字"

# 配置邮箱（建议和 GitHub 注册邮箱一致）
git config --global user.email "你的邮箱@example.com"
```

#### 示例

```bash
git config --global user.name "Zhang Wei"
git config --global user.email "zhangwei@company.com"
```

#### 验证配置是否成功

```bash
git config --list
```

看到 `user.name` 和 `user.email` 有对应的值，说明配置成功 ✅

> 💡 `--global` 表示全局配置，对你电脑上**所有 Git 项目**都生效，只需配置一次。

### 🐙 Step 3：注册 GitHub 账号

> 如果公司使用的是 **GitLab**，可跳过此步，直接使用公司账号。

1. 访问 [https://github.com](https://github.com)
2. 点击右上角 `Sign up`
3. 填写邮箱、密码、用户名，完成注册
4. 验证邮箱后登录 ✅

### 🔐 Step 4：配置 SSH 密钥（让电脑和 GitHub 互相认识）

每次向 GitHub 推送内容时，GitHub 需要验证你的身份。**SSH 密钥**就是你电脑和 GitHub 之间的"专属通行证"，配置一次，之后永久免密码操作。

> 💡 **类比**：就像门禁卡和门禁系统的关系。
> - 你的电脑生成一对"钥匙"：**私钥**（留在自己电脑上）+ **公钥**（交给 GitHub）
> - 每次连接时，GitHub 用公钥验证你的私钥，验证通过就放行

```
你的电脑                              GitHub
┌──────────┐                      ┌──────────────┐
│          │  ──── 出示公钥 ────▶  │  已登记你的   │
│  私钥    │                      │    公钥       │
│ (自己留) │  ◀─── 验证通过 ─────  │              │
└──────────┘                      └──────────────┘
```

#### 生成 SSH 密钥

打开终端 / Git Bash，输入：

```bash
ssh-keygen -t ed25519 -C "你的邮箱@example.com"
```

之后会出现三次提示，**全部直接按回车**（使用默认设置）：

```
Enter file in which to save the key: （回车）
Enter passphrase:                     （回车，不设密码）
Enter same passphrase again:          （回车）
```

看到类似下面的图案说明生成成功 ✅

```
+--[ED25519 256]--+
|        .o+      |
|       . o+o     |
|      . =.+..    |
|       = B +     |
|      . S = .    |
+----[SHA256]-----+
```

#### 查看并复制公钥

```bash
cat ~/.ssh/id_ed25519.pub
```

终端会输出一串以 `ssh-ed25519` 开头的文字，**全部复制**，类似：

```
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAI... 你的邮箱@example.com
```

#### 将公钥添加到 GitHub

1. 登录 GitHub，点击右上角头像 → `Settings`
2. 左侧菜单找到 `SSH and GPG keys`
3. 点击绿色按钮 `New SSH key`
4. `Title` 填写备注名（如 `我的MacBook` 或 `公司电脑`）
5. `Key` 栏粘贴刚才复制的公钥内容
6. 点击 `Add SSH key` ✅

#### 验证 SSH 连接是否成功

```bash
ssh -T git@github.com
```

看到以下输出，说明连接成功 🎉

```
Hi 你的用户名! You've successfully authenticated,
but GitHub does not provide shell access.
```

## 六、深入理解：暂存区、提交与撤销

### 🗂️ Git 的三个区域

在理解 `git add` 和 `git commit` 之前，先要知道 Git 把你的工作分成了**三个区域**：

```
┌─────────────┐    git add    ┌─────────────┐  git commit  ┌─────────────┐
│             │  ──────────▶  │             │  ──────────▶ │             │
│   工作区     │               │    暂存区    │              │   本地仓库   │
│ (你的文件夹) │               │  (待确认区)  │              │  (存档记录)  │
│             │               │             │              │             │
└─────────────┘               └─────────────┘              └─────────────┘
   你在这里修改文件               打包要提交的内容               正式存档！
```

> 💡 **生活类比**：
> - **工作区** = 你的书桌，东西摆在上面，随时可以改
> - **暂存区** = 一个托盘，把"这次要交出去的东西"放进去
> - **本地仓库** = 正式归档柜，放进去就有记录了

---

### 📌 git add —— 把改动放入"托盘"

`git add` 的作用是：**选择哪些改动要纳入本次提交**。

> 为什么需要这一步？因为你可能同时改了很多文件，但只想提交其中几个。

| 命令 | 含义 |
|------|------|
| `git add 文件名` | 只把这一个文件放入暂存区 |
| `git add .` | 把当前目录下**所有改动**都放入暂存区 |
| `git add *.md` | 把所有 `.md` 文件放入暂存区 |

> 💡 **图形化工具中的对应操作**：在 GitHub Desktop 中，左侧文件列表里每个文件前面都有一个**勾选框**，勾选 = `git add`，不勾选 = 本次提交不包含该文件。

---

### 📸 git commit —— 正式存档

`git commit` 的作用是：**把暂存区的内容正式存入本地仓库，生成一条版本记录**。

| 命令 | 含义 |
|------|------|
| `git commit -m "说明文字"` | 提交并附上说明（最常用）|
| `git commit --amend` | 修改**上一次**提交的说明文字 |

#### 示例

```bash
git add .
git commit -m "新增用户注册流程图，补充手机验证步骤"
```

> ⚠️ `git commit` 只是存到了**本地**，还没有上传到云端。需要再执行 `git push` 才能让团队看到。

---

### ↩️ 常用撤销命令

撤销是 Git 最实用的能力之一，根据"后悔的阶段"不同，有不同的命令：

```
┌──────────────┐         ┌──────────────┐         ┌──────────────┐
│    工作区     │         │    暂存区     │         │   本地仓库    │
│              │         │              │         │              │
│  改了但没 add │         │ add了但没     │         │  已经 commit  │
│              │         │ commit       │         │              │
└──────────────┘         └──────────────┘         └──────────────┘
       │                        │                        │
  git restore               git restore              git reset /
   文件名                   --staged 文件名           git revert
```

---

#### 情况一：改了文件，还没 add，想撤销修改

```bash
git restore 文件名
```

> ⚠️ **危险操作**：会直接丢弃工作区的修改，**无法恢复**，谨慎使用！

---

#### 情况二：已经 add，想把文件从暂存区移出来（保留修改内容）

```bash
git restore --staged 文件名
```

> 这个操作**不会丢失**文件改动，只是把它从"托盘"里拿回来，文件内容还在。

---

#### 情况三：已经 commit，想撤销提交

##### 方式 A：`git reset` —— "悄悄反悔"（修改历史）

```bash
# 撤销最近一次提交，改动退回到工作区（最常用）
git reset HEAD~1

# 撤销最近一次提交，但保留文件在暂存区
git reset --soft HEAD~1

# 撤销最近一次提交，同时丢弃所有改动（危险！）
git reset --hard HEAD~1
```

| 参数 | 暂存区 | 工作区文件 | 适用场景 |
|------|--------|-----------|----------|
| `--soft` | 保留 | 保留 | 只想改提交说明或重新整理提交 |
| `--mixed`（默认）| 清空 | 保留 | 想重新选择要提交的文件 |
| `--hard` | 清空 | **丢弃** | 彻底放弃这次改动 |

> ⚠️ `git reset` 会修改提交历史，**如果已经 Push 到远程，不建议使用**，会影响团队其他人！

---

##### 方式 B：`git revert` —— "光明正大地反悔"（新增一条撤销记录）

```bash
git revert HEAD
```

> `git revert` 不会删除历史，而是**新增一条提交**来抵消之前的改动。就像在文件里写"本条作废"，原来的记录还在。

| 对比 | `git reset` | `git revert` |
|------|-------------|--------------|
| 历史记录 | 会被修改/删除 | 保留，新增一条 |
| 适合场景 | 还没 Push 时 | 已经 Push 后 |
| 安全性 | 较危险 | 更安全 |

---

### 🗺️ 撤销场景速查表

| 我的情况 | 用这个命令 |
|---------|-----------|
| 改了文件，还没 add，想撤销 | `git restore 文件名` |
| 已经 add，想从暂存区移出 | `git restore --staged 文件名` |
| 已经 commit，还没 push，想撤销 | `git reset HEAD~1` |
| 已经 push 了，想撤销某次提交 | `git revert HEAD` |
| 想修改上一次的提交说明 | `git commit --amend` |

> 🎯 **记住一个原则**：
> - **还没 Push** → 可以用 `reset` 自由修改历史
> - **已经 Push** → 用 `revert`，不要动历史，保护团队协作

## 八、实际操作演示（以 GitHub Desktop 为例）

### Step 1：克隆仓库（第一次获取项目文件）

1. 打开 GitHub Desktop
2. 点击 `File` → `Clone Repository`
3. 输入仓库地址，选择本地保存路径
4. 点击 `Clone`，完成！

> 📌 之后这个文件夹就是你的"本地仓库"，里面的文件和普通文件夹一样，可以正常编辑。

---

### Step 2：拉取最新内容（每天开始工作前）

1. 打开 GitHub Desktop
2. 点击顶部的 `Fetch origin`（检查是否有新内容）
3. 如果有更新，点击 `Pull origin`（下载最新内容）

> ⚠️ **好习惯**：每次开始工作前先 Pull，避免和别人的修改冲突！

---

### Step 3：修改文件并提交

1. 在本地文件夹正常修改文件（改文档、改图片都可以）
2. 打开 GitHub Desktop，左侧会显示你修改了哪些文件
3. 勾选要提交的文件（相当于 `git add`）
4. 在底部 `Summary` 框里写上本次修改的说明，例如：`更新首页 Banner 设计稿`
5. 点击 `Commit to main`（存档！相当于 `git commit`）

---

### Step 4：推送到远程仓库

1. 提交后，点击顶部 `Push origin`（相当于 `git push`）
2. 完成！团队其他人现在可以看到你的改动了 🎉

## 九、常见问题 Q&A

**Q：Git 只能管理代码吗？**
> A：不是！Git 可以管理任何文本文件，包括 Markdown 文档、配置文件、设计规范文档等。不过对于 Word、PSD 等二进制文件，Git 能记录"有改动"，但无法直观对比具体改了什么。

---

**Q：我不小心改错了，怎么恢复？**
> A：这正是 Git 最强大的地方！在 GitHub Desktop 中，右键点击文件 → `Discard Changes`，即可恢复到上次提交时的状态。或者在历史记录中找到任意一个版本，直接回滚。

---

**Q：我和同事同时改了同一个文件，怎么办？**
> A：这叫"冲突"（Conflict）。Git 会提示你，需要手动选择保留哪个版本的内容。建议：
> - 修改前先 Pull
> - 尽量避免多人同时修改同一文件
> - 遇到冲突，找研发同学协助解决

---

**Q：提交说明（Commit Message）怎么写？**
> A：写清楚"做了什么"，让自己和别人一眼看懂。

| ❌ 不好的写法 | ✅ 好的写法 |
|-------------|------------|
| `update` | `更新用户注册流程图，增加手机号验证步骤` |
| `fix` | `修复首页文案错别字` |
| `改了一些东西` | `调整活动页 Banner 尺寸为 750x400` |

---

**Q：本地仓库和远程仓库有什么区别？**
> A：本地仓库在你自己电脑上，只有你能看到；远程仓库在服务器上（如 GitHub），团队所有人都能访问。`Push` 就是把本地同步到远程，`Pull` 就是把远程同步到本地。

---

**Q：遇到看不懂的报错怎么办？**
> A：直接把报错信息复制，问研发同学或者 AI（如 ChatGPT / Claude），基本都能快速解答，不用不好意思 😄

## 十、Git 使用规范建议

> 适合团队统一遵守的基本规范

1. **每次改动只做一件事**，不要把多个不相关的修改放在一次提交里
2. **提交说明要清晰**，用中文写明改动内容
3. **开始工作前先 Pull**，减少冲突
4. **不要直接修改主分支**（main/master），重要改动走分支 + 合并流程
5. **不要提交敏感信息**，如密码、密钥、内部数据等
6. **二进制大文件谨慎提交**，如高清设计稿、视频等，建议咨询研发同学

## 十一、总结

### 核心概念速查

| 概念 | 一句话理解 |
|------|-----------|
| 仓库（Repo）| 有记忆的文件夹 |
| 提交（Commit）| 给文件做存档 |
| 分支（Branch）| 平行宇宙，独立工作线 |
| 合并（Merge）| 把改动合并回主线 |
| 推送（Push）| 上传到云端 |
| 拉取（Pull）| 从云端同步到本地 |

### 日常操作三步走

```
        ⬇️ Pull          ✏️ 修改 + Commit          ⬆️ Push
   同步最新内容    →    本地编辑并存档    →    上传到远程仓库
```

### 撤销速查

| 我的情况 | 用这个命令 |
|---------|-----------|
| 改了文件，还没 add，想撤销 | `git restore 文件名` |
| 已经 add，想从暂存区移出 | `git restore --staged 文件名` |
| 已经 commit，还没 push，想撤销 | `git reset HEAD~1` |
| 已经 push 了，想撤销某次提交 | `git revert HEAD` |
